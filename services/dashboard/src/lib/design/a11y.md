# AĒR Accessibility Primitives

The design system commits to **WCAG 2.2 AA as a baseline**, with several
conventions that rise above AA where the scientific-instrument framing makes
it necessary (Design Brief §5.7 Progressive Semantics). This document
describes the primitives and the ARIA conventions every component is expected
to follow.

## Focus rings

- All interactive elements receive a visible focus ring via `:focus-visible`.
- The ring token (`--focus-ring-color`) is guaranteed to maintain at least
  **3:1 contrast against any background** in the token set (dark and light).
- Ring width is `--focus-ring-width` (2 px) with an offset of
  `--focus-ring-offset` (2 px) so the ring does not overlap glyph strokes.
- Never suppress the ring with `outline: none` without an equivalent visible
  replacement — doing so breaks keyboard navigation silently.

## Reduced motion

- `prefers-reduced-motion: reduce` collapses **all** motion-scale durations
  to `0.01ms` (see `tokens.css`). Animations that depend on a meaningful
  duration simply complete instantly.
- Auto-rotation (e.g. the 3D globe in Phase 99) must additionally check
  `matchMedia('(prefers-reduced-motion: reduce)')` at runtime and disable
  itself — CSS alone cannot disable imperative animation loops.

## Skip link

- Every page renders a single `<SkipLink>` as the first focusable element in
  DOM order.
- It is visually hidden until it receives keyboard focus (`:focus-visible`),
  then slides into view.
- Target: the main content region (`<main id="main">`). The ID is stable
  across all pages so deep-linked screen-reader users land consistently.

## ARIA conventions

- **Buttons.** Use `<button type="button">`. Never style an `<a>` or
  `<div>` as a button. If the action triggers a state change rather than
  navigation, the element is a button.
- **Dialogs.** Must declare `role="dialog"`, `aria-modal="true"`, and
  `aria-labelledby` referencing the heading. Focus must be trapped inside
  while open, and returned to the invoking element on close.
- **Tooltips.** Content is rendered inside an element referenced by
  `aria-describedby` on the trigger. The tooltip itself carries
  `role="tooltip"`. Hover alone is never sufficient — keyboard-focus also
  opens the tooltip.
- **Badges with status semantics.** When a badge carries methodological
  meaning (Epistemic Weight tier, validation state), the outer element
  declares `role="img"` and carries the human-readable text in
  `aria-label`; its visible label is wrapped in `<span aria-hidden="true">`.
  `aria-label` is prohibited on a bare `<span>` (axe
  `aria-prohibited-attr`), so the `role` is mandatory. Icon-only badges are
  forbidden.
- **Live regions.** Methodological refusal notices and descent-state
  announcements use `aria-live="polite"` so they reach screen readers
  without interrupting other speech.

## Color as the only distinguishing signal

Never. Every color-carried distinction has a redundant non-color encoding:

- Epistemic Weight tiers carry distinct line thickness, dash pattern, and
  text badge alongside the color stop.
- Status chips always render an icon (or a `::before` dot) in addition to
  the background color.
- Viridis-scaled encodings (density, time) are paired with axis labels and,
  where space allows, a legend with numeric stops.

## Testing

- `@axe-core/playwright` runs against every component story and every
  route. The build fails on any WCAG 2.2 AA violation (Phase 98c).
- Manual keyboard-only walkthrough is required for any new interactive
  component before merge.
- Screen-reader smoke-test with a real AT (NVDA on Windows, VoiceOver on
  macOS) before a surface lands in production.
