# 12. Glossary

| Term | Definition |
| :--- | :--- |
| **AĒR** | Ancient Greek ἀήρ — "the lower atmosphere, the surrounding climate." The project name, reflecting the system's purpose as an atmospheric sensor for societal discourse. |
| **Atmospheric Aesthetics** | The governing design concept of the AĒR dashboard: stratification, currents, stillness-with-motion, the observer at the edge, no faces. Every visual, interactive, and structural decision is evaluated against this principle. See [Design Brief §1](../design/design_brief.md). |
| **Authenticity Extractor** | A future extractor type proposed by WP-003 section 8.2 for detecting non-human actors (bots, coordinated campaigns) on platforms where user-generated content is present. Not applicable to RSS sources (Probe 0). Requires the `CorpusExtractor` path (R-9) for cross-document analysis. See R-12. |
| **At-Least-Once Delivery** | A messaging guarantee where the broker ensures every message is delivered to the consumer at least once. May result in duplicates, which AĒR handles via idempotency checks. Provided by NATS JetStream. |
| **Baseline** | The per-(metric, source, language) mean and standard deviation computed over a reference time window. Stored in `aer_gold.metric_baselines` and used as the denominator for z-score normalization. Computed offline by `scripts/compute_baselines.py`. See Phase 65 and WP-004 §5. |
| **BiasContext** | A Pydantic model (`models/bias.py`) that documents the structural biases of a data source's platform. Fields: `platform_type`, `access_method`, `visibility_mechanism`, `moderation_context`, `engagement_data_available`, `account_metadata_available`. Populated by source adapters during harmonization and carried in `SilverMeta` subclasses. Follows the WP-003 "document, don't filter" approach. See `docs/probes/probe-0-de-institutional-rss/bias_assessment.md` for the Probe 0 bias documentation. |
| **BFF (Backend-for-Frontend)** | An API layer (implemented in Go) that sits between the frontend and the analytical database (ClickHouse). It serves pre-aggregated, strictly typed data and shields consumers from direct database access. See `services/bff-api/`. |
| **Bronze Layer** | The first tier of the Medallion Architecture. Stores raw, unmodified data exactly as received from external sources in the MinIO `bronze` bucket. Data here is immutable (write-once) and expires after 90 days via ILM. |
| **Clean Architecture** | The internal directory structure of Go microservices: `cmd/` (entry point, zero logic), `internal/config/` (environment mapping), `internal/storage/` (infrastructure adapters), `internal/core/` (business logic with injected dependencies). |
| **Contract-First** | A design principle where the API specification (OpenAPI YAML) is written before the implementation. Go server stubs and types are generated from the spec via `oapi-codegen`. The spec is the SSoT for the API shape. |
| **Crawler** | A standalone, external program (under `crawlers/`) that fetches data from a public API and delivers it to AĒR via `POST /api/v1/ingest`. Crawlers are deliberately outside the system boundary — they follow the "Dumb Pipes, Smart Endpoints" pattern. |
| **Cultural Calendar** | A per-region YAML file under `configs/cultural_calendars/<region>.yaml` (Phase 68) enumerating public holidays, election dates, religious observances, and recurring major media events. Produced and maintained by researchers per WP-005 §4.3 and consumed as a static lookup when interpreting temporal metric patterns (e.g. distinguishing a holiday-driven volume drop from a genuine discourse shift). Probe 0 currently ships `de.yaml`. Not yet dynamically integrated into the query layer — the calendar exists to support manual interpretation. |
| **Content Catalog** | The BFF subsystem serving Dual-Register content (semantic + methodological text for metrics, probes, discourse functions, and refusal types) from YAML sources under `services/bff-api/configs/content/`. Introduced in Phase 95. Ensures that no Dual-Register copy is hardcoded in frontend source. See ADR-020 Layer 4 and [Design Brief §5.7](../design/design_brief.md). |
| **Dashboard** | The AĒR user-facing application. A static SvelteKit build serving three surfaces (Atmosphere, Function Lanes, Reflection) across five layers of progressive descent. Deployed behind Traefik on the `aer-frontend` network. See ADR-020 and the [Design Brief](../design/design_brief.md). |
| **Data Lake** | The collective term for AĒR's object storage system (MinIO), organized into Bronze, Silver, and Quarantine buckets. It holds all raw and harmonized data. |
| **Discourse Function** | One of four universal categories classifying a source's role in discourse production (WP-001 §3): Epistemic Authority (norm-setting through informational baseline), Power Legitimation (institutional agenda-setting), Cohesion & Identity (community cohesion and identity production), Subversion & Friction (counter-hegemonic discourse). Stored in `source_classifications` and propagated as `discourse_function` in the Gold layer. |
| **Dead Letter Queue (DLQ)** | The MinIO `bronze-quarantine` bucket. When the analysis worker fails to validate a document against the Silver Contract, the raw data is moved here instead of crashing the pipeline. Expires after 30 days via ILM. |
| **Deterministic Timestamp** | A timestamp extracted from MinIO event metadata (`event_time_str`) rather than the system clock (`datetime.now()` / `time.Now()`). Used for all ClickHouse inserts to ensure idempotency on NATS redeliveries. |
| **Docs-as-Code** | The practice of writing architecture documentation (Arc42) in Markdown, storing it in the same repository as the source code, and rendering it via MkDocs. Ensures documentation evolves alongside the codebase. |
| **Downsampling** | The BFF API's strategy of aggregating ClickHouse data into 5-minute intervals (`toStartOfFiveMinute`) with a hard row limit before returning results. Prevents OOM errors on large time-range queries. |
| **Epistemic Weight** | The visual prominence of a rendered metric, which scales with its methodological backing: Tier 1 unvalidated → moderate weight with status badge; Tier 2 with established validation → full weight. Weight is derived from live validation status (via `GET /api/v1/metrics/available`), never from a frontend constant. Ensures visual prominence scales with methodological merit. See [Design Brief §5.8](../design/design_brief.md) and ADR-016. |
| **Emic Tag** | A participant-perspective designation of a data source — how the source identifies itself or is identified within its own cultural context. Captures the local name (`emic_designation`), contextual self-understanding (`emic_context`), and language. Stored in `source_classifications`. Contrasts with the observer-imposed Etic Tag. See WP-001 §4.2. |
| **Etic Construct** | A universal, observer-defined analytical category (e.g., `evaluative_polarity`) to which one or more concrete metrics may be mapped. The `aer_gold.metric_equivalence` table links etic constructs to metrics with a validated equivalence level. Enables cross-cultural comparisons by abstracting over culture-specific measurement instruments. See WP-004 §4. |
| **Etic Tag** | An observer-imposed classification of a source's discourse function using the four universal categories of the Functional Probe Taxonomy. Applied by the research team during the probe classification process (WP-001 §4.4). Contrasts with the participant-perspective Emic Tag. See WP-001 §4.2. |
| **Durable Consumer** | A NATS JetStream subscription that persists its position in the message stream across consumer restarts. The analysis worker uses a durable consumer (`durable_name`) so no events are lost during deployments or crashes. |
| **Exponential Backoff** | A retry strategy where the wait time between attempts increases exponentially (1s, 2s, 4s, ...). Used by all Go services (via `cenkalti/backoff/v5`) for database and object storage connection attempts. Bound to the application's `context.Context` for clean shutdown. |
| **Gold Layer** | The third tier of the Medallion Architecture. Stores derived, aggregated numerical metrics in ClickHouse (`aer_gold.metrics`). This is what the BFF API queries. Data expires after 365 days via ClickHouse TTL. |
| **Go Workspace** | Go's native multi-module development feature (`go.work`). AĒR uses it to link `pkg/`, `services/ingestion-api/`, `services/bff-api/`, and `crawlers/wikipedia-scraper/` into a single development workspace, enabling local cross-module imports without publishing. |
| **Hard-Pinning** | The policy of specifying exact, immutable Docker image tags in `compose.yaml` (e.g., `postgres:18.3-alpine3.23` instead of `postgres:latest`). Ensures reproducible builds across machines and time. The compose file is the SSoT for all image versions. |
| **Harmonization** | The process of transforming raw, heterogeneous Bronze data into a clean, unified format (Silver). Includes whitespace normalization, field extraction, and word count calculation. Performed by the Python analysis worker. |
| **Healthcheck** | A Docker-level or application-level probe that determines if a service is operational. AĒR uses HTTP probes (e.g., `GET /healthz`) and native commands (e.g., `pg_isready`). Log parsing is strictly prohibited for healthchecks. |
| **IaC (Infrastructure as Code)** | The principle that infrastructure provisioning (bucket creation, schema initialization, stream setup) is defined in version-controlled scripts (`infra/`), executed by init containers — never by application code at runtime. |
| **Idempotency** | The property that processing the same input multiple times produces the same result as processing it once. In AĒR, achieved via PostgreSQL status checks (`processed` / `quarantined`) before re-processing a NATS event, and deterministic timestamps for ClickHouse inserts. |
| **ILM (Information Lifecycle Management)** | MinIO's built-in feature for automatic data expiration. AĒR uses ILM to delete Bronze data after 90 days and Quarantine data after 30 days. Configured in `infra/minio/setup.sh`. |
| **Ingestion Contract** | The source-agnostic JSON structure that every crawler must produce when submitting data to `POST /api/v1/ingest`. Contains `source_id`, and an array of `documents` each with a `key` (MinIO path) and `data` (raw payload). |
| **Init Container** | A Docker container that runs once at stack startup (`restart: "no"`) to provision infrastructure. AĒR uses `nats-init` (creates JetStream stream) and `minio-init` (creates buckets, applies ILM, registers NATS event notifications). |
| **Layer (Progressive Descent)** | One of five depth levels at which any dashboard surface can be viewed: Immersion (L0), Orientation (L1), Exploration (L2), Analysis (L3), Provenance (L4), Evidence (L5). Layers are orthogonal to Surfaces — every surface has all five layers. Descent deepens both hermeneutic access and cultural narrowing simultaneously. See [Design Brief §4](../design/design_brief.md). |
| **Liveness Probe** | An endpoint (`/healthz`) that returns HTTP 200 if the service process is alive. Does not check downstream dependencies. Used by Docker healthchecks and Kubernetes to determine if a container should be restarted. |
| **Macroscope** | The conceptual metaphor for AĒR: an instrument that reveals large-scale patterns invisible to the naked eye (analogous to a microscope for the small). AĒR is a "societal macroscope" observing global discourse patterns. |
| **Medallion Architecture** | A data engineering pattern that organizes data into three quality tiers: Bronze (raw), Silver (cleaned/harmonized), and Gold (aggregated/analytical). AĒR implements this across MinIO (Bronze/Silver) and ClickHouse (Gold). |
| **Metric Equivalence** | A validated assertion that a given metric, in a given language and source type, measures a universal etic construct at a specified level of comparability: `temporal` (only temporal patterns comparable), `deviation` (deviation from baseline comparable), or `absolute` (raw values directly comparable). Stored in `aer_gold.metric_equivalence`. Required for z-score normalization. See Phase 65 and WP-004 §6. |
| **Minimum Meaningful Window** | The finest temporal resolution at which a metric, given its source's publication rate, yields buckets that are statistically informative rather than mostly empty. Surfaced as `minMeaningfulResolution` on `GET /api/v1/metrics/available`. Initially seeded as a static BFF config map from Probe 0 publication-rate heuristics (tagesschau.de → `hourly`, bundesregierung.de → `daily`). Advisory only — the BFF does not enforce it. See Phase 66 and WP-005 §3.3. |
| **Metadata Index** | The PostgreSQL database that links MinIO object paths (`bronze_object_key`) to ingestion jobs, source IDs, and OpenTelemetry trace IDs. Enables Progressive Disclosure by providing the lookup layer between Gold metrics and Bronze raw data. |
| **Monorepo** | A repository structure where all services, shared libraries, infrastructure scripts, documentation, and API contracts reside in a single Git repository. AĒR uses this to keep everything synchronized. |
| **OLAP (Online Analytical Processing)** | A database paradigm optimized for read-heavy, aggregation-heavy queries over large datasets. ClickHouse is AĒR's OLAP engine, storing the Gold layer metrics. |
| **OpenTelemetry (OTel)** | A vendor-neutral observability framework for distributed tracing and metrics. All three AĒR services emit traces via OTLP gRPC to the OTel Collector, which fans out to Tempo (traces) and Prometheus (metrics). Trace context is propagated across NATS message headers. |
| **Partial Failure** | A scenario where one step of a multi-step operation succeeds but a subsequent step fails (e.g., MinIO upload succeeds but PostgreSQL insert fails). AĒR mitigates this via the `pending` → `uploaded` → `processed` status lifecycle in PostgreSQL. |
| **Probe Classification** | The five-step process by which a data source is formally classified within the Functional Probe Taxonomy (WP-001 §4.4): (1) area expert nomination, (2) peer review, (3) technical feasibility assessment, (4) ethical review, (5) registration. Results are stored in the `source_classifications` PostgreSQL table. Classifications with `review_status = 'provisional_engineering'` have not yet undergone this process. |
| **Progressive Semantics** | The Dual-Register communication pattern where every data point carries both a semantic register (plain language) and a methodological register (algorithm, tier, known limitations) in the DOM, with only one prominent at a time. The transition between registers is a local micro-interaction, never a page navigation. Semantic register is primary at Layers 0–3; methodological register is primary at Layer 4. See [Design Brief §5.7](../design/design_brief.md). |
| **Progressive Disclosure** | A UI/UX pattern where the dashboard initially shows aggregated trends (Gold) and allows analysts to drill down into the original raw data (Bronze) for qualitative verification. Enabled by the Metadata Index (PostgreSQL) linking Gold metrics back to Bronze objects via trace IDs. |
| **Readiness Probe** | An endpoint (`/readyz`) that returns HTTP 200 only if the service and all its critical dependencies (e.g., ClickHouse, PostgreSQL, MinIO) are operational. Used to determine if a service can accept traffic. |
| **Surface (Dashboard)** | One of three top-level encounter modes in the AĒR dashboard: Atmosphere (the 3D rotating globe), Function Lanes (discourse functions from WP-001 as horizontal lanes), and Reflection (methodological prose in the Distill.pub style). Surfaces are orthogonal to Layers — each surface has all five depth layers. Cross-context composition is not a fourth surface; multiple probes appear within the existing three surfaces. See [Design Brief §3](../design/design_brief.md). |
| **Sentinel Pattern** | The graceful shutdown strategy used by the Python analysis worker. Instead of cancelling running tasks, a `None` value (sentinel) is pushed into the `asyncio.Queue` for each worker, signaling them to finish their current task and exit cleanly. Prevents torn database connections. |
| **Probe 0 (German Institutional RSS)** | AĒR's first real data source — a set of German institutional RSS feeds selected for engineering pipeline calibration, not scientific representativeness. Sources include bundesregierung.de and tagesschau.de. See Chapter 13, §13.10 for full rationale and bias acknowledgment. |
| **RSS Crawler** | A standalone Go program (`crawlers/rss-crawler/`) that fetches, parses, and ingests RSS/Atom feeds into the AĒR pipeline. Configured via `feeds.yaml`. Maintains a local JSON state file for cross-run deduplication. Sets `source_type: "rss"` in the Bronze payload for downstream adapter resolution. |
| **RssMeta** | The `SilverMeta` subclass for RSS feed items. Contains `feed_url`, `categories`, `author`, and `feed_title`. Populated by the `RssAdapter` during harmonization. |
| **Schema Version** | An integer field (`schema_version`) in `SilverCore` that identifies the schema generation of a Silver record. Version 1 represents the legacy `SilverRecord` format (pre-Phase 39). Version 2 is the current `SilverCore` format. Enables the worker to handle multiple schema generations simultaneously in the append-only Silver bucket. See ADR-015. |
| **Silver Contract** | The two-tier Pydantic schema (`SilverCore` + `SilverMeta`) that defines the mandatory structure for harmonized data. Every Bronze document must pass validation against `SilverCore` before being promoted to the Silver layer. Documents that fail validation are routed to the DLQ. See ADR-015 for the schema evolution strategy. |
| **SilverCore** | The universal minimum contract for the Silver layer. Contains the fields every document must have for any NLP pipeline to operate: `document_id`, `source`, `source_type`, `raw_text`, `cleaned_text`, `language`, `timestamp`, `url`, `schema_version`, `word_count`. Changes to `SilverCore` require a formal ADR. See ADR-015. |
| **Silver Layer** | The second tier of the Medallion Architecture. Stores harmonized, schema-validated data in the MinIO `silver` bucket as `SilverEnvelope` objects (containing `SilverCore` + optional `SilverMeta`). Created by the analysis worker after successful Pydantic validation. Currently has no ILM expiration (see Chapter 11, Risk R-3). |
| **SilverMeta** | Source-specific metadata envelope attached to a Silver record. Each source type defines its own `SilverMeta` subclass (e.g., future `RssMeta` with `feed_url`, `author`, `categories`). Explicitly marked as unstable — adapters may restructure meta fields without a formal ADR. See ADR-015. |
| **Source Adapter** | A class implementing the `SourceAdapter` protocol that maps source-specific raw Bronze data to the universal `SilverCore` contract plus an optional `SilverMeta` envelope. Each data source (RSS, forum, legacy) provides its own adapter. Adapters are registered in the `AdapterRegistry` and looked up by `source_type`. See ADR-015. |
| **SSoT (Single Source of Truth)** | A design principle where exactly one authoritative source defines a piece of information. In AĒR: `compose.yaml` is the SSoT for container image tags, `.tool-versions` is the SSoT for developer tooling versions (golangci-lint, oapi-codegen, govulncheck, pip-audit), `openapi.yaml` is the SSoT for the BFF API contract, and `.env` is the SSoT for runtime configuration. |
| **Testcontainers** | A library (available for Go and Python) that spins up real, ephemeral Docker containers for integration testing. AĒR's Go tests use `testcontainers-go` against real PostgreSQL, MinIO, and ClickHouse instances. Python `test_storage.py` uses the Python equivalent. Both dynamically parse image tags from `compose.yaml` (SSoT enforcement). |
| **MetricExtractor** | A Python protocol (`extractors/base.py`) defining per-document metric extraction. Each implementation extracts one or more `GoldMetric` values from a single `SilverCore` record. Extractors are registered in `main.py` via dependency injection and executed sequentially by the `DataProcessor`. A failing extractor is skipped; other extractors continue (graceful degradation). |
| **Context Key** | A composite string (e.g., `de:rss:epistemic_authority`) that identifies the cultural and structural context in which a metric is measured. Used as part of the `metric_validity` table's composite key to scope validation results to specific language-source type-discourse function combinations. A metric may be valid in one context but not another. |
| **Metric Validity** | The degree to which a metric measures what it claims to measure, within a specific context. Stored in `aer_gold.metric_validity` with quantitative indicators: Krippendorff's alpha (`alpha_score`), correlation with reference annotations (`correlation`), sample size (`n_annotated`), and an expiration date (`valid_until`). See ADR-016 and WP-002 §4. |
| **Provisional Metric** | A metric produced by an extractor whose lexicon, model, or algorithm is an engineering default — not a scientifically validated choice. Phase 42 extractors (language detection, SentiWS sentiment, spaCy NER) are all provisional. Results should not be interpreted as validated measurements. The specific methods will be revisited when interdisciplinary collaboration (§13.5) provides methodological grounding. |
| **SentiWS** | SentimentWortschatz — a German-language word-level polarity lexicon published by Leipzig University under CC-BY-SA 4.0. Version 2.0 assigns positive or negative polarity weights to ~3,500 base forms plus inflections. Used by the `SentimentExtractor` (Phase 42, provisional). Does not handle negation, irony, or compositionality. |
| **Temporal Scale** | One of the five resolutions at which discourse phenomena unfold (WP-005 §3): event-scale (5 min), diurnal (hourly), weekly cycles (daily), seasonal (weekly), and multi-year drift (monthly). The BFF `?resolution` parameter selects which scale a `GET /api/v1/metrics` query is rendered at via the corresponding ClickHouse `toStartOf*` bucketing function. See Phase 66 and §8.13. |
| **Tiered Retention** | A retention strategy in which the raw 5-minute layer is kept only for the short term and progressively coarser pre-aggregated tables retain progressively longer history (target: 30 d full / 1 y hourly / 5 y daily / unbounded monthly). Defined as the target architecture in §8.8.1. **Status: planned — not yet active.** Activation depends on the deferred materialized views in migration `000009`. See Phase 66 and WP-005 §5.4. |
| **TTL (Time-To-Live)** | A database-level feature that automatically deletes rows after a specified duration. ClickHouse's `aer_gold.metrics` table uses a 365-day TTL on the `timestamp` column via the `MergeTree` engine. |
| **Validation Protocol** | The five-step process proposed in WP-002 §4 for establishing metric validity: (1) reference annotation by domain experts, (2) inter-annotator agreement (Krippendorff's alpha), (3) system-vs-reference correlation, (4) error taxonomy construction, (5) context-scoped validity declaration with expiration. Results are stored in `aer_gold.metric_validity`. |
| **Validation Study** | A concrete application of the Validation Protocol to a specific (metric, language, source_type, discourse_function) context. Each study is documented by a completed `docs/templates/validation_study_template.yaml` file (Phase 68) and produces a single row in `aer_gold.metric_validity`. Minimum methodological requirements: ≥ 3 annotators, Krippendorff's alpha ≥ 0.667. On expiration (`valid_until`) the corresponding metric reverts to `unvalidated` in `GET /api/v1/metrics/available`. See [Scientific Operations Guide → Workflow 2](../scientific_operations_guide.md#workflow-2-validating-a-metric). |
| **Visibility Mechanism** | The method by which a platform determines content ordering and exposure. Documented in `BiasContext.visibility_mechanism`. Examples: `chronological` (RSS, reverse-chronological feeds), `algorithmic` (social media recommendation engines), `curated` (editorially selected content). A key structural bias dimension (WP-003) because algorithmic visibility can amplify certain content types and suppress others. |
| **Methodological Transparency** | The WP-006 §6 principle that every metric must expose its algorithmic provenance — tier classification, algorithm description, known limitations, validation status, and extractor version — at the point of consumption, not buried in documentation. Implemented in Phase 67 via `GET /api/v1/metrics/{metricName}/provenance` backed by `configs/metric_provenance.yaml` plus dynamic lookups in `metric_validity` and `metric_equivalence`. See ADR-017. |
| **Non-Prescriptive Visualization** | The WP-006 §6 principle that visual encodings must not smuggle normative judgments into the display of metrics: no red/green for contested metrics, uncertainty alongside point estimates, multiple visualization modes, no hand-picked valence-laden palettes. Captured as frontend requirements in `docs/design/visualization_guidelines.md`. Deferred to the dashboard phase — not enforceable at the API layer. See ADR-017 Principle 2. |
| **Reflexive Documentation** | The WP-006 §6 principle that each data source must link to its own bias profile and classification rationale so consumers can audit the instrument, not just its outputs. Implemented in Phase 67 via the PostgreSQL `sources.documentation_url` column (migration 000007) and surfaced by `GET /api/v1/sources`. Phase 70 (migration 000008) repointed Probe 0 sources from a single bias-profile file to the probe dossier directory `docs/probes/probe-0-de-institutional-rss/`. See ADR-017 and §8.15. |
| **Probe Dossier** | A documentation pattern (Arc42 §8.15, Phase 70) that groups per-probe scientific context under `docs/probes/<probe-id>/` in five mandatory files: `README.md` (overview + WP coverage matrix), `classification.md` (WP-001), `bias_assessment.md` (WP-003), `temporal_profile.md` (WP-005), and `observer_effect.md` (WP-006). The PostgreSQL `sources.documentation_url` column points at the dossier directory rather than at any single file. WP-002 (validation) and WP-004 (equivalence) are deliberately *not* part of the dossier because they are system-wide rather than per-probe. The reference dossier is `docs/probes/probe-0-de-institutional-rss/`. |
| **Z-Score Normalization** | A statistical transformation that expresses a metric value as the number of standard deviations from its baseline mean: `z = (value - mean) / std`. In AĒR, used to make metrics comparable across sources and languages by removing scale differences. Available via `?normalization=zscore` on the BFF API `GET /api/v1/metrics` endpoint. Gated by baseline existence and equivalence validation. See Phase 65 and WP-004 §5. |
